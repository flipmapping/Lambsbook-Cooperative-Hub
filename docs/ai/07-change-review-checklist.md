# CHANGE REVIEW CHECKLIST — LAMBSBOOK

## PURPOSE
This checklist defines the mandatory review questions for any meaningful documentation change, code change, packet output, or AI-generated implementation result.

No meaningful change is accepted without review against this checklist.

---

## SECTION 1 — SCOPE CHECK

- Is the change bounded to the stated objective?
- Did it avoid unrelated cleanup or adjacent refactors?
- Did it touch only authorized files?
- Is the resulting diff proportionate to the objective?

If any answer is no:
- reject or narrow the change

---

## SECTION 2 — ARCHITECTURE CHECK

- Does the change preserve the canonical authority model?
- Does Gateway remain presentation/bridge only?
- Does Hub remain canonical domain orchestration authority?
- Does Supabase remain live backend/data truth?
- Does the change avoid creating a second truth system?

If any answer is no:
- stop and escalate to architecture correction

---

## SECTION 3 — CONTRACT CHECK

- Are runtime contracts unchanged unless explicitly intended?
- Are request/response meanings preserved?
- Are caller rules preserved?
- Are error semantics preserved?
- Was no implicit contract invented?

If any answer is no:
- reject until contract truth is clarified

---

## SECTION 4 — STATE-MACHINE CHECK

- Does the change preserve the canonical user states?
- Does it preserve the discovery flow?
- Does it preserve the materialize → accept distinction?
- Does it avoid optimistic domain completion?
- Does it keep 404 member-not-found semantics correct?

If any answer is no:
- reject until state-flow alignment is restored

---

## SECTION 5 — IDENTITY / AUTH CHECK

- Does the change preserve the canonical identity path?
- Does it avoid downstream identity re-resolution?
- Does it avoid email/member identity confusion?
- Does it preserve auth requirements of the affected routes?
- Does it avoid convenience weakening of access control?

If any answer is no:
- reject and inspect identity discipline

---

## SECTION 6 — EXECUTION QUALITY CHECK

- Was the work performed under a valid bounded packet?
- Were backups and recovery protections used where applicable?
- Was there any ambiguous or no-output execution?
- If yes, was inspect-before-rerun followed?
- Were unauthorized assumptions refused rather than filled in?

If any answer is no:
- do not accept the output yet

---

## SECTION 7 — VERIFICATION CHECK

- Was the diff inspected?
- Were the target files inspected directly?
- Was appropriate build/typecheck/test verification run?
- Was the result checked against architecture, contracts, and state flow?
- Is the change actually reviewable and understandable?

If any answer is no:
- verification is incomplete

---

## SECTION 8 — SYNCHRONIZATION CHECK

- Does docs/ai need updating?
- Does Open Brain need a capture because reusable truth changed?
- Does GitHub need a checkpoint commit?
- Are outdated truths now superseded and no longer authoritative?
- Is push safety understood if branch history changed?

If any answer is unclear:
- stop and complete synchronization before treating the change as settled

---

## SECTION 9 — REJECTION TRIGGERS

Reject immediately if the change:
- widens scope beyond packet
- creates parallel truth
- alters authority boundaries casually
- changes route semantics without approval
- collapses state distinctions
- bypasses governed backend truth
- introduces unreviewed hidden behavior
- mixes foundational docs and unrelated code without intent

---

## ACCEPTANCE RULE

A meaningful change is accepted only when:
1. scope is correct
2. architecture is preserved
3. contracts are preserved or intentionally updated
4. state-machine integrity is preserved
5. identity/auth discipline is preserved
6. verification is complete
7. synchronization duties are completed

---

## CANONICAL SUMMARY

Review is not optional.

No meaningful output becomes trusted merely because:
- it compiles
- it was generated by AI
- it was committed
- it appears to work locally

The change must satisfy the full review checklist.
