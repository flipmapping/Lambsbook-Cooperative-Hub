# CAR-001
Canonical Artifact Repository Convention

Execution Operating System: EOS v2.2

Status: Canonical

Materialized UTC: 2026-07-22T13:31:35+00:00

## Mission

Define the canonical repository convention by which all canonical
artifacts are deterministically resolved independently of registries
and independently of consuming implementations.

## Architectural Principle

Registries provide registration.

Repository conventions provide deterministic resolution.

Canonical artifacts provide semantic authority.

Consumers derive interpretations.

## Canonical Authority Chain

Canonical Identifier
        │
        ▼
Registry
        │
Registration
        ▼
Repository Convention
        │
Resolution
        ▼
Canonical Artifact
        │
Meaning
        ▼
Consumer

## Initial Specialization

Execution Contracts.

Future artifact categories shall adopt this convention rather than
introduce artifact-specific repository rules.

## Constraints

Implementations shall not:

- hard-code filesystem layouts;
- infer repository structure;
- embed artifact paths.

Repository resolution shall derive exclusively from this convention.

## Canonical Artifact Resolution Strategy

### Architectural Contract

Canonical artifact resolution is a two-stage process.

1. Resolve a canonical identifier.
2. Resolve that identifier into the authoritative canonical artifact.

Consumers shall depend only upon this contract.

### Current Repository Convention

The current repository provides the first realization of the
resolution strategy.

Future repository organizations may differ provided they continue
to satisfy the architectural contract defined above.

### Execution Contract Specialization

For Execution Contracts the strategy defines:

- canonical identifier format;
- artifact category;
- deterministic resolution algorithm;
- current canonical repository realization;
- permitted artifact representations;
- version selection policy;
- deprecation policy;
- deterministic failure behaviour.

### Consumer Responsibilities

Consumers shall:

- request canonical identifiers;
- invoke the published resolution strategy;
- consume only canonical artifacts.

Consumers shall not:

- infer repository topology;
- embed repository paths;
- reinterpret repository conventions.


## Execution Contract Repository Realization

The current repository realization for canonical Execution Contracts is
defined independently of consuming implementations.

### Artifact Category

Execution Contract

### Canonical Identifier

EC-<identifier>

### Repository Root

execution/contracts/

### Canonical Artifact Location

execution/contracts/canonical/

### Artifact Naming

<canonical_identifier>.md

Example:

EC-VAL-001
    ↓
execution/contracts/canonical/EC-VAL-001.md

### Resolution Rule

Given a canonical identifier:

1. Validate registration.
2. Apply this repository realization.
3. Resolve the canonical artifact.
4. Load the canonical artifact.
5. Derive semantics exclusively from that artifact.

Future repository realizations may change provided they continue to
satisfy CAR-001.
